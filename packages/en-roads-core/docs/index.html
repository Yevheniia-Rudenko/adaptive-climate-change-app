@climateinteractive/en-roads-core #

This package is the primary entry point to the En-ROADS SDK for most users. It includes two main classes:

  • the Model class, which simplifies the process of running the model when inputs change and refreshing the user interface when outputs are updated
  • the Config class, which provides “specs” that describe how model inputs and outputs can be displayed in a user interface

See the en-roads-sample-web and en-roads-sample-node apps included with the En-ROADS SDK for detailed usage examples.

En-ROADS Model API #

High-level Model #

  • Functions:
    • createAsyncModel: Use this to create a Model that schedules and runs the Wasm model asynchronously on a worker thread.
  • Types:
    • Model

Model inputs #

  • Functions:
    • createModelInput: Use the InputSpec objects provided by the Config to create Input instances that can be used with the Model or ModelRunner types.
  • Types:
    • Input
      • SliderInput
      • SwitchInput
    • InputCallbacks

Model outputs #

  • Functions:
    • createDefaultOutputs: Use this to create an Outputs instance that is initialized to hold output data produced by a ModelRunner (most applications won’t need to use this).
  • Types:
    • Outputs
      • Series
      • Point
    • SocialCostOutputs

Advanced: Low-level ModelRunner #

  • Notes:
    • The low-level ModelRunner interface is used in the implementation of the high-level Model class. Most applications don’t need to interact with it directly, but it is exposed as a public API for advanced use cases.
  • Functions:
    • createAsyncModelRunner: Use this to create a ModelRunner that runs the Wasm model asynchronously on a worker thread.
    • createDefaultModelRunner: Use this to create a ModelRunner that runs the Wasm model synchronously on the main JavaScript thread (this is not recommended, but may be useful in advanced use cases).
  • Types:
    • ModelRunner

En-ROADS UI Configuration API #

Core Config entry point #

  • Functions:
    • getDefaultConfig: Use this to get the default Config instance that provides access to all “spec” types that can be used to create a user interface around the En-ROADS model.
  • Types:
    • Config

Input specs #

  • Related properties:
    • Config#inputs: Use the inputs map to access the InputSpec instances, which correspond to the input variables that are exposed by the En-ROADS model.
  • Types:
    • InputSpec
      • SliderSpec
      • SwitchSpec

Policy input specs #

  • Related properties:
    • Config#policyGroups: Use the policyGroups map to access the PolicyGroupSpec instances, which describe a set of model inputs in a particular policy area.
  • Types:
    • PolicyGroupSpec: Describes a group of policy inputs as displayed in an “Advanced” view.

Assumption input specs #

  • Related properties:
    • Config#assumptionGroups: Use the assumptionGroups map to access the AssumptionGroupSpec instances, which describe a set of model inputs in a particular grouping of assumptions.
  • Types:
    • AssumptionGroupSpec: Describes a group within the Assumptions view.

Graph specs #

  • Related properties:
    • Config#graphs: Use the graphs map to access the GraphSpec instances, which correspond to the graphs that are preconfigured to display output variable data from the En-ROADS model. (You can also define your own custom GraphSpec instances as long as the referenced variables are exposed by the En-ROADS model.)
  • Types:
    • GraphSpec
      • GraphDatasetSpec
      • GraphLegendItemSpec
      • GraphAlternateSpec
      • GraphSide
      • GraphKind
      • UnitSystem
      • HexColor
      • LineStyle
      • LineStyleModifier

Data table specs #

  • Related properties:
    • GraphSpec#dataTable: There are a small number of special GraphSpec instances that do not describe a normal graph, but instead display a table of raw figures. If the dataTable property is defined, you can access the DataTableSpec that describes the data that appears in the table.
  • Types:
    • DataTableSpec
      • DataTableRowSpec
      • DataTableValueId

Outcome specs #

  • Related properties:
    • Config#outcomes: Use the outcomes array to access the OutcomeSpec instances, which correspond to the outcomes that are used to display key indicators (such as those in the “Actions & Outcomes” view in En-ROADS).
  • Types:
    • OutcomeSpec

Other common types #

  • Identifier types:
    • InputVarId
    • OutputVarId
  • String types:
    • StringKey
    • FormatString
